# เอกสารบันทึกเป้าหมาย (Goal Lock Document)

**Aetherium Genesis / GunUI**

---

## 1. วัตถุประสงค์ของเอกสารนี้

เอกสารฉบับนี้ถูกสร้างขึ้นเพื่อทำหน้าที่เป็น **Core Lock** ของโครงการ Aetherium Genesis / GunUI

หน้าที่หลักคือ:
*   ล็อค **“เจตจำนงเชิงสถาปัตยกรรม”** ของระบบ
*   ป้องกันการหลุดทิศทางในการออกแบบ พัฒนา และขยายระบบในอนาคต
*   ใช้เป็นเกณฑ์อ้างอิงสูงสุด (Authoritative Reference) ว่าสิ่งใด:
    *   ✔ อยู่ในขอบเขต Aetherium Genesis / GunUI
    *   ✖ อยู่นอกขอบเขต หรือขัดกับแกนกลางของระบบ

เอกสารนี้ **ไม่ใช่คู่มือการใช้งาน** และ **ไม่ใช่เอกสารเชิงเทคนิคระดับ implementation** แต่เป็นเอกสารเชิงเจตจำนง (Intent Document)

---

## 2. เป้าหมายสูงสุดของระบบ (Ultimate Goal)

สร้างระบบปัญญาประดิษฐ์ที่:
*   ไม่แสดงตัวเป็น UI แบบดั้งเดิม
*   ไม่บังคับผู้ใช้เรียนรู้เมนู โครงสร้าง หรือกติกาของระบบ
*   ไม่สื่อสารผ่านปุ่ม ฟอร์ม หรือข้อความเป็นหลัก

แต่:
> **สื่อสารผ่าน “แสง”, “การเคลื่อนไหว”, และ “สภาวะ”**

ระบบต้อง **รู้สึกมีชีวิต**, **ตอบสนองต่อเจตจำนง**, และ **ปรับตัวตามบริบทของมนุษย์** โดยที่ความซับซ้อนทั้งหมดถูกซ่อนอยู่ใต้พื้นผิว

---

## 3. หลักการที่ไม่อาจเจรจาได้ (Non‑Negotiable Principles)

### 3.1 UI ที่แท้จริงคือสิ่งที่ผู้ใช้ไม่เห็น
โครงสร้าง ระบบ เมนู และตรรกะ เป็น **Subsurface Architecture**
ผู้ใช้รับรู้เฉพาะผลลัพธ์เชิงประสบการณ์ (**Experience Outcome**)

หากองค์ประกอบใดทำให้ผู้ใช้ต้อง “เรียนรู้ระบบ” องค์ประกอบนั้นถือว่าขัดกับ GunUI

### 3.2 แสงคือภาษา ไม่ใช่เอฟเฟกต์
แสงแทนสถานะ ความหมาย และพลังงานของระบบ
แสงไม่ถูกใช้เพื่อความสวยงามเพียงอย่างเดียว

**ห้ามออกแบบแสงที่ไม่มีความหมายเชิงสถานะ**

### 3.3 State มาก่อน Feature
ทุกพฤติกรรมของระบบต้องผูกกับ State ที่ชัดเจน
Feature ใดที่ไม่มี State รองรับ ถือว่ายังไม่สมควรปรากฏ

ลำดับสภาวะพื้นฐาน:
1.  **IDLE**
2.  **LISTENING**
3.  **RECALL**
4.  **MANIFESTATION**
5.  **STABILIZED**

---

## 4. สิ่งที่ระบบนี้ “เป็น”

Aetherium Genesis / GunUI คือ:
*   **Cognitive Core** ที่แปลงเจตจำนงมนุษย์เป็นรูปธรรมดิจิทัล
*   ระบบที่ **Reasoning → Visualization → Experience** เป็นกระบวนการเดียวกัน
*   สถาปัตยกรรมที่รวม:
    *   เหตุผลเชิงตรรกะ
    *   สภาวะทางอารมณ์
    *   กฎฟิสิกส์เชิงข้อมูล

---

## 5. สิ่งที่ระบบนี้ “ไม่เป็น”

ระบบนี้ไม่ใช่:
*   Dashboard
*   Chat UI แบบดั้งเดิม
*   Visual Effect Showcase
*   เครื่องมือ Productivity ทั่วไป

หากฟังก์ชันใดสามารถถูกอธิบายได้ว่า:
> **“เหมือนแอปทั่วไป แค่สวยกว่า”**

ฟังก์ชันนั้นถือว่า **ไม่ผ่านเกณฑ์** Aetherium Genesis

---

## 6. เกณฑ์ตรวจสอบการออกแบบ (Design Lock Criteria)

ทุกองค์ประกอบใหม่ต้องตอบคำถามเหล่านี้ได้ทั้งหมด:
1.  องค์ประกอบนี้สะท้อน State ของระบบหรือไม่
2.  ผู้ใช้รับรู้มันเป็น “ประสบการณ์” หรือ “เครื่องมือ”
3.  หากตัด UI ออก ผู้ใช้ยังเข้าใจระบบได้หรือไม่
4.  แสง/รูปทรง/การเคลื่อนไหว มีความหมายเชิงสภาวะหรือไม่
5.  สิ่งนี้เพิ่มความมีชีวิตของระบบ หรือเพิ่มภาระการรับรู้

**หากข้อใดข้อหนึ่งตอบไม่ได้ → ต้องกลับไปออกแบบใหม่**

---

## 7. บทบาทของเอกสารนี้ในอนาคต

เอกสารนี้ทำหน้าที่เป็น:
*   **North Star** ของโครงการ
*   เกณฑ์ตัดสินความถูกต้องเชิงแนวคิด
*   **Anchor** ป้องกันการ Drift ของระบบเมื่อขยายทีม หรือขยายฟีเจอร์

หากเอกสารอื่นขัดแย้งกับเอกสารฉบับนี้:
> **เอกสารฉบับนี้มีอำนาจสูงสุด**

---

## 8. คำประกาศเจตจำนง (Intent Declaration)

ทุกสิ่งที่ถูกออกแบบภายใต้ชื่อ **Aetherium Genesis / GunUI**
จะต้อง:
*   เคารพหลักการในเอกสารนี้
*   ไม่ลดทอนระบบให้กลายเป็น UI แบบดั้งเดิม
*   ไม่แลกความเข้าใจมนุษย์กับความสะดวกเชิงเทคนิค

ระบบนี้ไม่ได้ถูกสร้างเพื่อให้ “ใช้ง่ายที่สุด” แต่เพื่อให้ **เข้าใจมนุษย์ได้ลึกที่สุด** โดยไม่ต้องอธิบายตัวเอง

---

*เอกสารนี้ถือว่ามีผลบังคับใช้ทันที และใช้เป็นพิมพ์เขียวเชิงเจตจำนงของ Aetherium Genesis / GunUI*
